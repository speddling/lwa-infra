Homelab Journal

Production discipline.
Residential address.

Christmas 1981 brought a TI-99/4A and the beginning of something that never really stopped. A few decades of ISP work, carrier NOC operations, cloud infrastructure, and DevOps later, the itch to have a real lab at home has never left. This is mine... and I treat it exactly like I treat production.

The Context

A real place. A real lab.

I was writing BASIC, drawing sprites, and sharing cassette-tape games with local user groups before most people knew what a personal computer was. High school at brand-new Wallkill Valley Regional in Hamburg, NJ brought BASIC, machine language, Pascal, and FORTRAN. The head of the computer lab, Mr. Zimmerman, told me programmers were going to be glorified secretaries and the real money was in hardware design. So I pivoted. The world did not.

The correction came when I found out our local BBS had purchased an internet gateway. I upgraded my Apple IIc to a Performa 6200CD and figured out the rest as I went. ISP work, carrier NOC operations, cloud infrastructure, DevOps... and somewhere along the way, I ended up back at university, pursuing a BS in Computer Science at Maestro University, with a focus on AI. The lab isn't separate from that path. It's part of it.

Little Wolf Acres is 2.3 acres in rural New Jersey. The household includes my wife, our daughter, two Kangals (Brutus and Junia, people-sized house dogs with strong opinions about the property line), two beagles (Soul and Maka, the latter better known online as Makamakamelon), and Cooper the rescued tuxedo cat. At some point the house also acquired a proper homelab. It started, as these things do, with one server. Then a second machine for DNS. Then monitoring. Then GitOps. Then three MCP servers so my AI assistant could read logs and open pull requests.

That escalated appropriately.

The goal was never just to have stuff running. The goal was to build infrastructure the same way I build it at work: everything in code, nothing clicked through a UI and forgotten, changes tracked and reviewed, documentation kept current. If I wouldn't do it in production without a PR, I don't do it here without one either. The lab grows as I grow. Right now that means getting more serious about AI infrastructure, local inference, and where the industry is heading. School is changing how I see the landscape, and the lab is where I work it out in practice.

The honest pitch to technical people: this is a portfolio in infrastructure form. Every config file, Ansible role, Kubernetes manifest, and GitHub Actions workflow lives in the public homelab repo with commit history going back to the first line. If I put this level of rigor into a home network that serves a family and their very large dogs, you can imagine what I bring to the job.

The page you're reading is the story of how it's all built, what's running, and where it's going. The short version lives here; the deep technical specs are in the expandable sections throughout.

There's something satisfying about purpose-built infrastructure. Each machine in this lab has a clearly defined role and doesn't do anything outside it. Monolith handles compute. Watchtower handles DNS and monitoring. Apex is the control plane and my daily workstation. Studio is the DAW that doesn't need to know any of this exists.

The Network

Managed switches, recursive DNS, and ad-free browsing for the whole house.

The network runs on TP-Link Omada gear: a multi-WAN VPN router, a cloud controller, and two access points covering the main house and the yarn studio. Everything is fully managed, SNMP-monitored, and assigned static IPs by MAC address binding at the router. Nobody sets a static IP at the OS level.

DNS is the part I'm probably most opinionated about. Every query from any device on the network goes to Watchtower first, which runs AdGuard Home. AdGuard filters ads, trackers, and malicious domains, then hands off to Unbound, which does recursive resolution directly against the root name servers. No upstream DNS provider. No query logging leaving the property. Every internal hostname resolves under littlewolfacres.com so you can reach anything by name.

The whole family gets ad-free browsing, tracker blocking, and DNSSEC validation as a side effect of me wanting a proper DNS stack. Not the worst outcome.

The Cluster

Kubernetes at home. Because why not, and also because GitOps is addictive.

Monolith runs k3s, which is a lightweight Kubernetes distribution designed to run on hardware that isn't a cloud region. Single-node by design, but the architecture is fully expandable if I ever add another machine.

Everything that runs on the cluster is managed by ArgoCD, a GitOps controller. ArgoCD watches the homelab repository and continuously reconciles what's running on the cluster against what's committed in the repo. I don't kubectl apply anything by hand. If it's not in the repo, it doesn't exist.

TLS certificates are handled automatically by cert-manager, which issues and renews Let's Encrypt certificates via Cloudflare DNS-01 challenge. No inbound ports required. Services that need HTTPS get it without any manual certificate work.

The Minecraft story (it's a good one)

The server spent months in a broken state because the original setup used a LoadBalancer service type for UDP traffic. k3s ServiceLB silently fails for UDP, so the server started fine but nobody could connect. After diagnosing that, switching to NodePort, fixing the world-import initContainer, setting up operator permissions for both players, and adding DNS, the server was live.

Then I built a Slack bot (the Zombatron Importer) that runs as a launchd service on Apex. Drop a .mcworld file in the private #zombatron Slack channel, the bot downloads it, SCPs it to Monolith, asks for confirmation, and on "yes" triggers a GitHub Actions workflow that clears the import marker and bounces the pod. Uses Socket Mode so no inbound ports needed. The world export from the $8/month Realm subscription is pending, at which point we cancel the subscription.

Observability

If it runs in this house, Prometheus knows about it.

The monitoring stack lives on Watchtower and watches everything: both compute nodes, the network gear, the DNS server, the NVR, and even the T-Mobile 5G gateway feeding the house internet. Every alert goes to a private Slack channel. Every dashboard is provisioned from disk by Ansible and will be deleted if it wasn't put there by code.

Twice a day, at 8 AM and 8 PM, a systemd timer fires a Python script that queries Prometheus, formats the results into a human-readable summary, and posts it to the #sentinel channel. The family sees "all systems good" or "storage getting full." I see exactly which PromQL query fired the alert.

Both custom exporters are original work. The T-Mobile FAST 5688W gateway doesn't expose a Prometheus endpoint, so I wrote one. Same with the Reolink NVR. Both are deployed via Ansible roles and scraped by Prometheus like any other target.

AI Tooling

Three MCP servers let my AI assistant read the cluster, write to the repo, and check the logs.

The most unusual part of this lab is the AI integration layer. I've built three MCP (Model Context Protocol) servers that give Claude structured, scoped access to the homelab. This is what made it possible to build most of what you see on this page entirely in conversation, with Claude doing the actual git work.

The design constraint I was strict about: each server has the minimum permissions needed for its job and nothing else. Synapse can read the cluster but can't change anything. Scribe can write to the repo but only through branches and pull requests, never directly to main. Argus can read monitoring configs and logs but has no write surface at all.

Synapse
Claude's eyes on the cluster. Read-only access to k8s pod state, Prometheus queries, Alertmanager alerts, and the Monolith filesystem within defined paths.
monolith:30800
Scribe
Claude's git control plane. Branch, commit, push, and open pull requests against this repo. Refuses to operate on main. All mutations require human review and merge.

Read-only access to the monitoring layer: live Alertmanager and Prometheus configs, systemd service state, journald logs, and monitoring HTTP APIs on Watchtower.

All three are FastMCP servers running as system services (launchd on Apex, systemd on Watchtower, Kubernetes on Monolith). Claude Desktop connects to them via Streamable HTTP. UFW restricts each to Apex's IP only.

Alongside the MCP layer, Apex runs B-4, an Ollama instance using the M4's unified memory for local LLM inference via Metal backend. It's the starting point for AI/ML workflows until Lore (a dedicated Mac Mini M4 Pro with 48GB) can be added later this year. School is the context: I'm at Maestro University studying CS with an AI focus, and the lab is where theory meets the actual hardware.

How Changes Happen

Everything goes through a pull request. No exceptions.

This is the part I'm most deliberate about. A homelab can very easily become a pile of undocumented changes that you're afraid to touch. Mine works exactly like a production environment: every change is a branch, the branch becomes a pull request, the PR gets reviewed, and merge triggers whatever needs to happen next.

GitHub Actions runs on self-hosted runners on Monolith and Watchtower. Ansible handles configuration management across all nodes. Terraform Cloud manages the infrastructure-as-code state. The result is a lab where I can look at the git history and understand exactly what changed, when, and why.

On the Roadmap

What comes next (and why it isn't done yet).

A few things are on the near-term list, mostly in priority order:

RAM upgrade for Monolith. 32GB is fine now but getting into AI inference workloads locally will want more headroom. 64GB is cheap for DDR4 at this point.

UPS coverage. The NUT role is written and waiting. The hardware budget just hasn't landed on a CyberPower unit yet.

Lore. A Mac Mini M4 Pro with 48GB of unified memory, headless, running Ollama as a dedicated inference node for the LAN. B-4 on Apex handles it for now. Lore is later this year.

Loki. Log aggregation on Watchtower. The Prometheus stack is solid; logs are the missing piece of the observability triad.

Managed switch. The unmanaged PoE switch gets replaced with a JetStream unit that can expose per-port SNMP stats. Then Prometheus can see everything the network is actually doing.
